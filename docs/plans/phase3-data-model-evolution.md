# Phase 3: Data Model Evolution

## Overview

Add missing fields to event schemas and alerts that are needed for future algorithm development, client-level isolation, and audit tracking.

**Source:** bugs.md #7, #13; changes.md #10, #11, #13

## Changes

### 3.1 Add `additional_fields` to all event types

**Why:** bugs.md #7 — "add an additional_fields field to all event types that contains a json with values that can be used in future algos"

**ClickHouse schema** (`scripts/clickhouse_init.sql`):

Add to `orders`, `executions`, `transactions` tables:
```sql
additional_fields String DEFAULT '{}'
```

Stored as a JSON string in ClickHouse (String column, not a native JSON type). Algorithms and queries can parse it as needed.

**Enrichment** (`de_platform/pipeline/enrichment.py`):

Pass through `additional_fields` from raw event if present, default to `'{}'`.

**Normalizer validation** (`de_platform/pipeline/validation.py`):

`additional_fields` is optional. If provided, must be a dict (validated, then serialized to JSON string).

### 3.2 Add `client_id` to all event types

**Why:** changes.md #10 — "all of our events have (or should have) a client_id which represents the actual user of the tenant that is doing the trade/transaction. Algos should take that field into account and we should use that field in order to create test isolation"

**ClickHouse schema** — add to `orders`, `executions`, `transactions`:
```sql
client_id String DEFAULT ''
```

Also add to `duplicates` and `normalization_errors` tables for traceability.

**Enrichment**: Pass through `client_id` from raw event. Required field — validation should warn (not reject) if missing for backward compat.

**Algorithm impact**: VelocityAlgo suppression key changes from `tenant_id` to `(tenant_id, client_id)` (connects to Phase 1 velocity suppression). Other algos receive it in event dict for future use.

**Test isolation**: E2E tests can use `client_id` for finer-grained isolation within a shared tenant. This is a future enhancement — current tenant-based isolation remains.

### 3.3 Add `ingestion_method` to all event types

**Why:** changes.md item in Data Audit Redesign #11 — "events should contain an ingestion method field (file, rest, kafka) to use for the batching"

**ClickHouse schema** — add to `orders`, `executions`, `transactions`:
```sql
ingestion_method String DEFAULT ''
```

Values: `"file"`, `"rest"`, `"kafka"`

**Set by ingestion modules**:
- `kafka_starter` → sets `ingestion_method = "kafka"`
- `rest_starter` → sets `ingestion_method = "rest"`
- `file_processor` → sets `ingestion_method = "file"`

Each module adds the field before publishing to normalization topics.

### 3.4 Add `event_ids` to alerts

**Why:** changes.md #13 — "alerts should have a list of event ids. We need to use the event ids in the relevant e2e tests to make sure that alerts/cases created are really the ones that should have been created"

Currently alerts have a single `event_id` field. For window-based algos like VelocityAlgo, an alert represents a condition across multiple events. We need to capture all contributing event IDs.

**Postgres alerts table** — migration:
```sql
ALTER TABLE alerts ADD COLUMN event_ids TEXT[] DEFAULT '{}';
```

**Algorithm changes** (`de_platform/pipeline/algorithms.py`):
- Add `event_ids: list[str]` field to `Alert` dataclass
- `LargeNotionalAlgo`: `event_ids = [event["id"]]` (single event)
- `VelocityAlgo`: `event_ids = [e["id"] for e in events]` (all events in window)
- `SuspiciousCounterpartyAlgo`: `event_ids = [event["id"]]` (single event)
- Keep `event_id` (singular) as the "trigger" event for backward compat

**Alert manager**: Persist `event_ids` alongside `event_id`. Use `event_ids` for cross-reference in case aggregation Rule 2 (same event overlap).

**E2E test impact**: Alert assertions can verify that the exact event IDs generated by the test are present in the alert's `event_ids` list.

## Acceptance Criteria

1. All event tables have `additional_fields`, `client_id`, `ingestion_method` columns
2. Ingestion modules set `ingestion_method` correctly
3. Events with `additional_fields` and `client_id` pass through enrichment/normalization
4. Alert dataclass has `event_ids` list, populated by all algorithms
5. Alert manager persists `event_ids`
6. Existing tests continue to pass (new fields are optional with defaults)
7. Unit tests cover new field validation and enrichment
